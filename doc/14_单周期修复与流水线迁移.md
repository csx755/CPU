## 1. 单周期 CPU 的 Bug 修复

在单周期 RV32I 实现过程中，共发现并修复了以下 bug：

### 1.1 LW 指令控制信号错误（commit `818b8a2`）

**问题**：LW 指令的 RegWrite、ALUSrc、WDSel 三个控制信号不正确，导致 Load 数据无法正确写回寄存器。

**根因**：`ctrl.v` 中 `itype_l` 的译码条件未正确包含所有 Load 指令（LB/LH/LW/LBU/LHU），控制信号表格中 Load 类指令的映射遗漏。

**修复**：
```verilog
// 修正前：部分 Load 指令的 RegWrite=0（不写回）
// 修正后：所有 itype_l 指令 RegWrite=1, ALUSrc=1, WDSel=FromMEM
assign RegWrite = rtype | itype_r | itype_l | utype_lui | utype_auipc | i_jal | i_jalr;
assign ALUSrc   = itype_r | itype_l | stype | utype_lui | utype_auipc;
assign WDSel[0] = itype_l;  // FromMEM
```

### 1.2 JALR 操作码错误（Phase 2）

**问题**：JALR 指令使用 OP-IMM (0x13) 操作码而非正确的 JALR (0x67)。

**根因**：`emit_JALR` 汇编器函数调用了 I-type 编码，I-type 的 opcode 固定为 `0b0010011` (OP-IMM)，但 JALR 的操作码应该是 `0b1100111`。

**修复**：新增 `JALR_enc()` 编码函数，opcode = 0x67。
```python
def JALR_enc(rs1, rd, imm=0):
    return ((imm & 0xFFF) << 20) | ((rn(rs1) & 0x1F) << 15) | \
           (0 << 12) | ((rn(rd) & 0x1F) << 7) | 0b1100111
```

### 1.3 x0 寄存器写保护（commit `48ce758`）

**问题**：可向 x0 写入非零值，违反 RISC-V 规范（x0 硬连线为 0）。

**根因**：`RF.v` 的写操作对所有寄存器一视同仁，未检查目标寄存器是否为 x0。

**修复**：
```verilog
// 修正前
rf[A3] <= WD;
// 修正后
rf[A3] <= (A3 != 0) ? WD : rf[A3];  // x0 不可写
```

| 修复项 | 文件 | 严重程度 |
|--------|------|:---:|
| LW 控制信号 | `ctrl.v` | 🔴 致命 |
| JALR 操作码 | 汇编器 | 🔴 致命 |
| x0 写保护 | `RF.v` | 🟡 中等 |
| ALU C 值锁存 | `alu.v` | 🟡 中等（综合警告） |

## 2. 架构对比：单周期 vs 流水线

### 2.1 核心差异

```
单周期 CPU：                        5 级流水线 CPU：
┌─────────────────────────┐         ┌─────┐ ┌─────┐ ┌─────┐ ┌─────┐ ┌─────┐
│ IF→ID→EX→MEM→WB         │         │ IF  │→│ ID  │→│ EX  │→│ MEM │→│ WB  │
│ (1 条指令/周期)           │         │     │ │     │ │     │ │     │ │     │
│ 最长路径决定时钟频率       │         │PC+4 │ │译码 │ │ALU  │ │DM   │ │RF写 │
│ 无冒险                   │         │取指 │ │读RF │ │转发 │ │     │ │MUX  │
└─────────────────────────┘         └──┬──┘ └──┬──┘ └──┬──┘ └──┬──┘ └─────┘
                                      │IF/ID  │ID/EX  │EX/MEM │MEM/WB
                                      │寄存器  │寄存器  │寄存器  │寄存器
                                      └───────┘└───────┘└───────┘
```

| 特性 | 单周期 | 流水线 |
|------|--------|--------|
| CPI | 1.0 | ~1.0 (理想) |
| 每周期工作 | 取指+译码+执行+访存+写回 | 各阶段独立并行 |
| 时钟周期 | 由最慢指令决定 | 由最慢单级决定 |
| 控制信号 | 组合逻辑直接驱动 | 随指令逐级传递 |
| 数据冒险 | 不存在 | 需转发 + 阻塞 |
| 控制冒险 | 不存在 | 需冲刷 |
| 模块复用率 | — | ctrl/alu/EXT/dm 完全复用 |

### 2.2 模块变化一览

| 模块 | 单周期 | 流水线 | 变化 |
|------|:---:|:---:|------|
| `PC.v` | ✅ | ✅ | **不变** |
| `ctrl.v` | ✅ | ✅ | **不变** |
| `alu.v` | ✅ | ✅ | **不变** |
| `EXT.v` | ✅ | ✅ | **不变** |
| `dm.v` | ✅ | ✅ | **不变** |
| `RF.v` | ✅ | ✅（微调） | x0 写保护 |
| `NPC.v` | ✅ | ❌ 移除 | NPC 逻辑内联到 SCPU_pipelined |
| `SCPU.v` | ✅ | ❌ 替换 | 被 SCPU_pipelined.v 替代 |
| `GRE_array.v` | — | 🆕 | **新增**：流水线寄存器 |
| `Hazard_Unit.v` | — | 🆕 | **新增**：冒险检测 |
| `Forwarding_Unit.v` | — | 🆕 | **新增**：数据前递 |

## 3. 流水线 CPU 新增模块

### 3.1 GRE_array.v — 通用流水线寄存器

**文件**：`code/rtl/GRE_array.v`

**功能**：参数化位宽的流水线寄存器，支持：
- **write_enable**：冻结（Load-Use 阻塞时保持 IF/ID）
- **flush**：清零/插入气泡（分支跳转时冲刷）
- **异步复位**（posedge Rst）

```verilog
module GRE_array #(parameter WIDTH = 64)(
    input  Clk, Rst, write_enable, flush,
    input  [WIDTH-1:0] in,
    output reg [WIDTH-1:0] out
);
    always @(negedge Clk or posedge Rst) begin
        if (Rst)          out <= 0;
        else if (flush)   out <= 0;
        else if (write_enable) out <= in;
    end
endmodule
```

**设计决策**：
- **negedge Clk**：参考教师模板，数据和 PC（posedge）错开半个周期，保证时序
- **合并单 always 块**：避免 Vivado 综合报多驱动 DRC MDRV-1 错误

**四个实例**：

| 实例 | WIDTH | 位置 | 包含信号 |
|------|:-----:|------|------|
| IF_ID | 64 | IF→ID | instruction + pcplus4 |
| ID_EX | 224 | ID→EX | 全部控制信号 + RD1/RD2 + imm + PC + rd/rs1/rs2 |
| EX_MEM | 108 | EX→MEM | RegWrite + ALU_result + RD2 + rd + DMType |
| MEM_WB | 104 | MEM→WB | RegWrite + DM_dout + ALU_result + rd + pcplus4 |

### 3.2 Hazard_Unit.v — 冒险检测单元

**文件**：`code/rtl/Hazard_Unit.v`

**功能**：纯组合逻辑，检测 Load-Use 冒险和控制冒险，控制流水线停顿和冲刷。

```verilog
// Load-Use: ID/EX 是 Load, 且 rd 与 ID 级 rs1/rs2 冲突
load_use_hazard = ID_EX_MemRead &&
    ((ID_EX_rd == ID_rs1 && ID_rs1 != 0) ||
     (ID_EX_rd == ID_rs2 && ID_rs2 != 0));

// IF/ID 冲刷: JAL 冲掉目的地址不在 IF 的错误指令
IF_ID_flush = ID_is_JAL && (IF_PC != ID_jal_target);

// ID/EX 冲刷: Load-Use(插气泡) 或 EX 分支跳转
ID_EX_flush = load_use_hazard | EX_taken;

// IF/ID 冻结: Load-Use 保持当前指令
IF_ID_write = ~load_use_hazard;
```

**关键修复**（commit `b8a5be6`）：
```verilog
// 修正前：JAL 无条件冲 IF/ID → 冲掉跳转目标本身
IF_ID_flush = ID_is_JAL;
// 修正后：只有当 IF 取的指令不是跳转目标时才冲
IF_ID_flush = ID_is_JAL && (IF_PC != ID_jal_target);
```

### 3.3 Forwarding_Unit.v — 数据前递单元

**文件**：`code/rtl/Forwarding_Unit.v`

**功能**：纯组合逻辑，检测 RAW 冒险，控制 ALU 输入的多路选择。

**前递路径**：
```
ForwardA/B = 00 → ID_EX.RD1/RD2（无转发，默认）
ForwardA/B = 01 → EX_MEM_ALU_result（最新结果，优先级高）
ForwardA/B = 10 → WB_WD（次新结果，优先级低）
```

**优先级规则**：EX/MEM > MEM/WB — 最新结果优先。
```verilog
// EX/MEM 优先级更高
ex_hazard  = EX_MEM_RegWrite && (EX_MEM_rd != 0) && (EX_MEM_rd == ID_EX_rs1);
mem_hazard = MEM_WB_RegWrite && (MEM_WB_rd != 0) && (MEM_WB_rd == ID_EX_rs1)
             && ~ex_hazard;
```

## 4. 流水线寄存器设计

### 4.1 信号逐级传递

**核心理念**：控制信号在 ID 级一次性译码，随指令逐级流动到需要的级。

```
ID 级译码 → ID/EX → EX/MEM → MEM/WB → RF
          RegWrite  RegWrite  RegWrite
          MemWrite  MemWrite
          ALUOp               ALU_result
          ALUSrc              DM_dout
```

**不需要的信号在传递中丢弃**：如 ALUOp 只到 EX 级，MemWrite 只到 MEM 级。

### 4.2 写入/冲刷控制

| 场景 | IF_ID.write | IF_ID.flush | ID_EX.flush | EX_MEM.flush |
|------|:---:|:---:|:---:|:---:|
| 正常流动 | 1 | 0 | 0 | 0 |
| Load-Use 阻塞 | **0** | 0 | **1** | 0 |
| 分支跳转 (EX 决议) | 1 | 1 | 1 | 0 |
| JAL (ID 决议) | 1 | 1* | 0 | 0 |

> \* JAL: 仅当 IF_PC ≠ ID_jal_target 时才冲（commit `b8a5be6`）

## 5. 数据通路修改

### 5.1 PC 更新路径（IF 级）

```
单周期：NPC 模块 → PC
        ├── NPCOp[2:0] 控制：+4 / BranchTarget / JumpTarget / JALR_target
        └── 复杂的多路选择

流水线：内联 NPC 逻辑
        IF_NPC = EX_taken  ? EX_branch_target :
                 ID_is_JAL ? ID_jal_target   :
                             IF_PC + 4;
```

**NPC.v 被移除**，逻辑内联到 SCPU_pipelined.v 中，更简洁。

### 5.2 写回路径（WB 级）

```
单周期：WD_MUX ← ALU_out / DM_dout / PC+4 → RF.WD (同周期)

流水线：WD_MUX ← MEM_WB.ALU_result / MEM_WB.DM_dout / MEM_WB.pcplus4
        ↓
        RF.WD (比译码晚 3 周期)
        RF.A3 ← MEM_WB.rd (写回目标来自 WB 级)
```

### 5.3 访存路径（MEM 级）

```
单周期：Addr_out = ALU 当前输出 (组合逻辑)
        Data_out = RF.RD2 (当前读)
        mem_w    = ctrl.MemWrite (当前译码)

流水线：Addr_out = EX_MEM.ALU_result (MEM 级锁存)
        Data_out = EX_MEM.RD2 (EX/MEM 中传递)
        mem_w    = EX_MEM.MemWrite
```

> **对外接口完全兼容**：外部模块（MIO_BUS/dm_ctrl）看到的端口时序完全相同，内部只是来源从"当前"变成了"MEM 级锁存值"。

## 6. 冒险与解决方法

### 6.1 数据冒险：前递（Forwarding）

```
场景：addi x1, x2, 5    ← x1 在 EX 级算出
      addi x3, x1, 10   ← x1 在 EX 级就需要！

解决：ForwardA = 01 (EX_MEM → EX)
      addi x3, x1(来自转发), 10 ✓
```

**转发路径覆盖**：
- EX/MEM → EX：前一条指令的结果
- MEM/WB → EX：前两条指令的结果
- Store 转发：ForwardB 也用于 Store 数据的转发

### 6.2 数据冒险：Load-Use 阻塞

```
场景：lw   x1, 0(x2)    ← x1 到 MEM 级才就绪
      addi x3, x1, 5    ← EX 级就需要 x1 — 转发来不及！

解决：插入 1 周期气泡
      IF_ID.write = 0 (冻结)
      ID_EX.flush = 1 (NOP)
```

**为什么转发无法解决？**
Load 数据在 MEM 级末（negedge clk）才存到 EX/MEM 寄存器，但依赖指令在 EX 级 posedge（半个周期前）就需要数据。时序上无法转发，只能阻塞一周期。

### 6.3 控制冒险：分支预测

**策略**：静态预测不跳转（PC+4），猜错时冲刷。

**JAL（ID 级决议）**：
```
代价：1 cycle（IF/ID 中 1 条错误指令被 flush）
JAL x1, target    IF    ID   (JAL 译码 → flush IF_ID)
addi (错误)       IF    ID   (被冲掉)
target: xxx                IF    ID   EX   ...
```

**分支（EX 级决议）**：
```
代价：2 cycles（IF/ID + ID/EX 共 2 条被 flush）
BEQ x1,x2,target  IF    ID    EX   (分支成立 → flush 前两级)
addi (错误)       IF    ID    (冲掉)
or   (错误)            IF    (冲掉)
target: xxx                     IF    ID    EX   ...
```

**JALR（特殊处理）**：
```
JALR 在 EX 级决议，触发 EX_taken 冲 ID/EX
已取到 IF/ID 的错误指令用 NOP 填充吸收（已知限制，见 §9）
```

## 9. 已知限制

### 9.1 JALR IF/ID 冲刷

**问题**：JALR 在 EX 级决议，触发 EX_taken 冲刷 ID/EX，但 IF/ID 的错误指令**不会被硬件自动冲刷**。

**当前方案**：在 JALR 指令后插入 2 条 NOP（软件层面），吸收已取入 IF/ID 的错误指令。

```asm
jalr x1, x0, 0(x5)   # 跳转
nop                    # 吸收 IF/ID 中的错误指令
nop                    # 确保下一条从正确的 PC 取指
```

**曾探索的硬件修复**（已回退）：
```verilog
// Hazard_Unit 增加 EX_is_JALR 输入
IF_ID_flush = ID_is_JAL | EX_is_JALR | EX_taken;  // 会导致测试失败
```
该硬件修复引入兼容性问题，已回退（详见 commit `3d5378b`）。

### 9.2 行为仿真兼容性

**问题**：Vivado XSim 行为仿真中，negedge GRE_array + posedge PC 的混合沿设计导致 X 传播。

**解决**：使用后综合仿真（post-synthesis functional simulation），综合后的门级网表无此问题。iVerilog 行为仿真正常。

### 9.3 ALU C 值锁存（仅综合警告）

**问题**：`alu.v` 中 `C` 寄存器在 case 不完全时产生锁存器推断警告。

**影响**：仅 Vivado 综合警告，不影响功能。C 值用于分支条件判断，正常工作。
